## 3) Обчислення та оновлення KPI
- **Cron `flf_performance_kpi_today` (за замовчуванням кожні 2 хв; конфіг.)**:
  - Перераховує тільки «сьогоднішні» дані для out/in.
  - Інкрементальний перерахунок: брати `stock.picking` з `write_date > last_run` і станами {done, assigned, confirmed}, оновлювати відповідні зрізи у `flf.performance.kpi`.
  - Зберігати `last_run` у `ir.config_parameter` (`flf_performance.last_kpi_run`).
  - Інтервал зберігати у `ir.config_parameter` (`flf_performance.refresh_interval_sec`).
  - Статтеринг: застосовувати зсуви оновлення за партнером/користувачем, щоб уникати піків.
  - Вести `last_updated_at` по кожному блоці/партнеру для відображення часу останнього оновлення у UI.
  - Для кожного слоту `flf.truck.slot` обчислювати: `created_since_prev_slot`, `shipped_in_slot`, `missed_in_slot`; показувати на Wallboard і в персональних панелях за приналежністю до партнера.
- **Метрики**:
  - `pickings_done`: кількість `stock.picking` у Done.
  - `lines_done`: сума рядків (`stock.move.line`) фактично виконаних для Done-пікингів.
  - `qty_done`: сума `qty_done` з `stock.move.line` для Done-пікингів.
  - `pct_in_deadline`: `done_в_дедлайн / done_всього`.
  - `avg_sec_print_to_done`: середній `(date_done - x_flf_print_ts)` по пікингам, де є обидві мітки.
  - `pickings_risk_count`: активні не-Done, дедлайн сьогодні, час до дедлайну < 60 хв.
  - Додатково: `done_in_soft_window`, `done_in_last_truck`, `done_afterhours_transfer`.
  - «Консолідація зусиль»: базові евристики — ріст створених заявок vs ковзне середнє; зростання заявок на одного активного співробітника; відсутність активності окремих працівників протягом 1 години від старту зміни (визначаємо як «можливо відсутній»). Пороги конфігуруються.
  - «Відправки у вихідні» (`offday_shipments`): лічильник по компанії та по співробітнику, використовується у бонусних показниках і дашбордах.

## 4) Інтерфейси та візуалізації
- **Меню та дії**:
  - Inventory → Operations → FLF Performance → `Wallboard` (client action, повноекранний, авто-оновлення 120с).
  - Inventory → Reporting → FLF Performance KPIs → список/канбан/графік/півод для `flf.performance.kpi`.
  - Inventory → Operations → `At Risk Pickings Today` → дерево `stock.picking` з доменом ризику.

  - Inventory → FLF Performance → Configuration:
    - `Settings` — глобальні налаштування через конфіг‑майстер (безпеки/ACL як у `res.config.settings`):
      - Пороги «консолідації зусиль»: `consolidation_x_percent` (40), `consolidation_y_percent` (30), `consolidation_window_days` (14).
      - Гнучка субота: пороги `sat_backlog_orders_{11,10,09}`, `sat_backlog_lines_{11,10,09}`, `sat_created_today_by_10`, а також дефолтні години старту за днями тижня.
      - Інтервал автооновлення та базовий offset для статтерингу.
      - Відображення/кредит «off‑day shipments» (увімк/вимк) і правила компанійного бонусу «все створене поїхало».
    - `Partner SLA Registry` — централізований реєстр партнерів у вигляді редагованого списку `res.partner` з FLF‑полями (`x_flf_deadline_hour`, `x_flf_soft_window_minutes`, `x_flf_last_shipment_hour`, `x_flf_afterhours_credit`, `x_flf_big_order_line_count`, `x_flf_big_order_qty`, `x_flf_use_draft_trigger`, `x_flf_responsible_user_ids`). Підтримати inline‑редагування (editable="bottom").
      - Мета: адміністратор складу керує усіма SLA/порогами в одному місці, без переходу у форму партнера.
      - Додаткові фільтри: «Партнери з особливими SLA», «Активні/неактивні».
    - `User Assignments` — централізований список `flf.user_config` (колонки: `company_id`, `user_id`, `primary_partner_id`), призначення «primary partner» (масові дії — у backlog).

  - Політика UI для форм:
    - Додати вкладку «FLF Performance» на формі партнера з полями SLA для швидкого доступу; основний менеджмент — через `Partner SLA Registry`.

### Гібридна конфігурація (погоджено)
- **Компанійні глобальні параметри**: зберігаємо як company‑dependent `ir.config_parameter` через `res.config.settings` (тільки UI у нашому `Settings`; форму компанії не модифікуємо).
- **Користувацькі налаштування**: окрема модель `flf.user_config` з полями `company_id`, `user_id`, `primary_partner_id` (+ інші FLF‑атрибути за потреби). Журнал змін увімкнено через `mail.thread`/`mail.activity.mixin` (chatter).
- **SLA партнерів**: керуємо лише через `Partner SLA Registry` (табличний список). Форму партнера не модифікуємо (вкладки не додаємо). За потреби — окремий візард масових змін (у backlog).
- **Доступ**: усі екрани Configuration доступні лише групі `flf_performance.group_manager` (FLF Performance Manager); Admin бачить усе.
- **Multi‑company**: усі параметри — строго company‑dependent; `flf.user_config` містить `company_id` для розмежування записів.
- **Wallboard (службовий акаунт на великому моніторі)**:
  - Блок «Сьогодні»: загальні метрики по компанії.
  - Рядки по партнерах: done/всього, % у дедлайні, risk count, прогрес-бар з кольорами: зелений/жовтий (<60 хв)/червоний (протерміновано).
  - ТОПи співробітників (3 категорії):
    1) за кількістю: окремо `lines_done` і `qty_done`;
    2) за швидкістю: найменший середній час на одну заявку та на одну одиницю;
    3) за педантичністю: найвищий відсоток попадання у дедлайн.
  - Список «У зоні ризику» (перші N, сортовані за часом до дедлайну).
  - Авто-оновлення через JS (setInterval) або через `bus`/RPC.
  - Показувати біля кожного блоку «Оновлено: HH:MM:SS».
  - Блок «Відправки по слотах»: для кожного найближчого слоту — скільки створено від попереднього слоту, скільки відвантажено, скільки не поїхало (missed). Показувати також per‑partner.
- **Персональна панель співробітника**:
  - Завжди видимий «HUD‑стріп» у верхній панелі між systray‑віджетами та меню: показує персональні ключові метрики за сьогодні (малі лічильники з кольорами).
  - Правий вузький док у «Складі» — не згортається (стислий режим), для тих самих метрик + підказки; деталі/історія відкриваються по кліку на елемент HUD або іконку systray.
  - Додаткове меню (fallback): Inventory → My Performance Today.
  - Додатково: показник «скільки створених не поїхало у слот» за закріпленим партнером працівника (відповідальність розподіляється між усіма відповідальними за партнера).
  - Бейдж «Off‑day» (коли співробітник робив відправки у вихідний/неділю), з підрахунком `offday_shipments`.
- **Вбудовані візуальні індикатори**:
  - У списку `At Risk Pickings Today` показувати бейджі статусів (✅/⚠️/❌) і «велике замовлення».

### Frontend (OWL/JS)
- Пакет активів у `web.assets_backend`: systray‑віджет, Wallboard client action (OWL), легкі RPC до модельних методів/контролерів.
- Конфігурація автооновлення (`flf_performance.refresh_interval_sec`) + розсинхронізація оновлень (offset) на фронті.
 - Ніяких важких SPA: мінімальні компоненти, що не заважають стандартному UX.
 - Відображення часу останнього оновлення у кожному блоці; невеликий відлік до наступного оновлення.
 - Рядки партнерів: показувати стани (не роздруковано / роздруковано не зібрано / Done / прострочено), час до дедлайну, waiting‑метрики, перелік виконавців за сьогодні з лічильниками.
 - ТОПи за періодами (день/тиждень/місяць): за товарами, рядками, замовленнями; фільтр періоду замість жорсткого Today.

## 5) Сповіщення (мінімальна версія)
- Внутрішні повідомлення (Discuss) менеджеру/каналу, коли у партнера `risk_count > 0` і до дедлайну < 30 хв.
- Обмеження частоти (не частіше 1 раз/год/партнер).
- Telegram — у наступній черзі (опційно).
- У текстах повідомлень відображати, чи вкладаємось у «мʼяке вікно» та «останню машину», щоб коректно пріоритезувати.
 - Додаткові «бонусні» нотифікації: коли «все створене сьогодні відправлено» (на рівні компанії).

## 6) Права доступу та безпека
- Групи:
  - `flf_performance.group_user` — перегляд власної міні-панелі.
  - `flf_performance.group_manager` ("FLF Performance Manager") — ОКРЕМА група, яка спадкується від `stock.group_stock_manager` і надає доступ до розділу Configuration (Settings, Partner SLA Registry, User Assignments, Truck Schedule). Адміністратор Odoo має доступ до всього незалежно від груп.
- ACL/record rules для `flf.performance.kpi` лише для внутрішніх юзерів.

- Multi-company:
  - Усі ключові параметри зберігаємо як company‑dependent (`ir.config_parameter` через `res.config.settings`).
  - Налаштування виконуються окремо для кожної компанії; меню Configuration показуються в контексті поточної компанії.
  - Cron обробляє дані per‑company (один cron, що ітерує по компаніях, або окремі crons — вирішимо на етапі реалізації; за замовчуванням один cron із ітерацією).
  - `flf.user_config` і пов’язані реєстри мають `company_id` і record rules по компанії.

## 7) Продуктивність і індекси
- Індекси в БД на `stock.picking(state, picking_type_code, partner_id, company_id, date_done, x_flf_print_ts, x_flf_deadline_at)`.
- `flf.performance.kpi`: індекси по вимірах.
- Обмежити запити «сьогоднішнім» інтервалом у часовому поясі користувача.

## 8) Етапи впровадження
1. Скелет модуля, групи, меню, пусті в’юхи, дані доступу.
2. Поля на `res.partner` (без UI у формі) для SLA; редагування виключно через `Partner SLA Registry`.
2a. Конфігурація в одному місці (Inventory → FLF Performance → Configuration):
   - `Settings` (конфіг‑майстер для `ir.config_parameter`).
   - `Partner SLA Registry` (редагований список `res.partner` з FLF‑полями, фільтри).
   - `User Assignments` (список `res.users` з колонкою `x_flf_primary_partner_id`).
2b. Гібрид: 
   - Включити chatter і затвердити company‑dependent для всіх параметрів.
   - Створити `flf.user_config` (з `company_id`, `user_id`, `primary_partner_id`, `mail.thread`).
   - Перенести «User Assignments» на `flf.user_config` (табличне редагування); у формах користувача — нічого не додаємо.
   - Вилучити/не використовувати вкладку «FLF Performance» на формі партнера; керувати через `Partner SLA Registry`.
3. Поля/логіка на `stock.picking`: атрибуція Done‑юзера, дедлайн від `create_date`, ознаки `in_soft_window`/`in_last_truck`/`afterhours_transfer`. Фіксація друку — повторне використання існуючого модуля.
4. Cron + інкрементальний перерахунок `flf.performance.kpi` для «сьогодні» (out/in) + конфіг. інтервал і статтеринг.
5. Frontend (OWL/JS): HUD‑стріп у хедері, systray‑деталі, Wallboard client action; JSON‑ендпоїнти; позначення часу оновлення.
6. Wallboard: список ризиків + 3 категорії ТОПів (кількість/швидкість/педантичність).
7. Персональна панель: HUD‑стріп (завжди видимий) + правий док (незгортуваний) у «Складі».
8. Менеджерська панель/дашборд з фільтрами за періодами + стандартні звіти (півод/графіки).
9. Мінімальні сповіщення (Discuss) із врахуванням soft/last‑truck/afterhours та «все створене відправлено».
10. Тести, фікси продуктивності, документація.
 11. Backlog на потім: майстри масових змін для Partner SLA Registry (пакетне оновлення дедлайнів/порогів за фільтрами/тегами партнерів).

## 9) Відкриті питання до узгодження
- Вирішено: дедлайн від `create_date`; дефолт 17:00; incoming — аналіз у 2‑й черзі.
- Вирішено: атрибуція «хто зробив Done» — достатньо на 1‑му етапі.
- Вирішено: інтервал автооновлення — конфігурабельний; застосовуємо статтеринг.
- Вирішено: Top‑5 — два окремих рейтинги (за рядками і за qty).
 - Вирішено: три категорії ТОПів — кількість/швидкість/педантичність.
 - Вирішено: дефолти «великого» замовлення — 50 рядків / 150 одиниць.
 - Вирішено: `x_flf_afterhours_credit = True` за замовчуванням.
 - Вирішено: пороги «консолідації зусиль» — X=40%, Y=30%, вікно ковзного середнього 14 днів (конфіг.).
 - Вирішено: субота старт 12:00 за замовчуванням; перемикання на 09:00 якщо протягом перших 30 хв «консолідація зусиль» > X два поспіль 15‑хв інтервали; завжди доступний ручний оверрайд менеджера.
 - Вирішено: розподіл `missed_in_slot` між відповідальними — у v1 повний кредит (100% кожному відповідальному); v2 — опційно зважено за часткою `lines_done` у слоті.

## 10) План перевірки/тестування
- Демо-дані: кілька партнерів із різними дедлайнами, out/in пікинги у різних станах.
- Юніт-тести: обчислення дедлайну, фіксація друку, атрибуція Done, агрегації KPI.
- Навантажувальна перевірка: «сьогодні» 5–20 тис. рядків — час оновлення KPI < 10 с.
