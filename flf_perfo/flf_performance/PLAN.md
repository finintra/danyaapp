# модуль flf_performance

## Цель

Создать модуль в odoo15, который будет мониторить работу сотрудников фулфилмента, держать сотрудников в ритме и одновременно контролировать выполнение обязательств перед партнёрами.

## Некоторые исходные данные
Каждый сотрудник (за исключением грузчиков) работает под своим аккаунтом. Каждый партнер - это компания в Оду15.
Сотрудники работают в основном в модуле Склад, у всех всегда открыта и периодически заходят в основную панель Склада. Это в основном сборщики заказов и приемщики.
Есть обязательство перед партнерами - заявки (отгрузки out) должны ехать день в день, если созданы партнерами до определенного часа. У одних это 17:00, у других 16:00, этот дедлайн разный. У некоторых партнеров дополнительно есть другие критерии скорости, например "большая заявка в день". Большая - это например, где больше "х" строк с товарами. Или больше чем "у" единиц товаров.
Возможно, в процессе реализации модуля ты дашь советы, что еще и как мне важно мониторить, что поможет ускорить сотрудников. 
Нужно наглядно выводить, укладываются ли в дедлайн сегодня. Сколько кто обработал штук товаров сегодня. Или строк заказов. Важно мониторить и приемки.
Сотрудники распечатывают заказы (отгрузки или приемки) с общего аккаунта для печати, и с этого момента это "их" заказ (но пока не привязан к конкретному сотруднику). Привязка происходит в момент, когда сотрудник начинает менять статус заказа в системе. Обычно это изменение со статуса Ready в статус Done. 

## Практическая реализация

К ноутбуку подключается дополнительный большой монитор и на него выводится на отдельном служебном аккаунте информация по процессам. В двух больших залах стоит по одному такому ноутбуку с монитором. Дополнительно у каждого сотрудника на своем ноутбуке на своем аккаунте свой небольшой блок с его показателями, подсказками и сигналами. Автообновление информации каждые 2 минуты по сегодняшнему дню (?).


## Особенности работы модуля

1. Основные показатели для онлайн-панели (wallboard)
1.1. Общий прогресс по дню по всем сотрудникам вместе, по каждому партнеру (партнер = company), по каждому сотруднику(у каждого на своем ноутбуке): 

- Количество обработанных заявок (out) за день.

- Количество строк (lines) в обработанных заявках.

- Количество единиц товара (qty_done).

- Процент заявок, уложившихся в дедлайн.

- Среднее время с распечатки до Done.

- Данные относительно дня, недели, месяца.

Это даст визуальное чувство темпа: «успеваем / отстаём».

1.2. Разрез по сотрудникам

- Общее количество обработанных строк / товаров за день.

- Среднее время на одну строку.

- Время с распечатки до Done.

- Среднее отклонение от нормы (норма задаётся индивидуально).

- Количество ошибок (например, отмен или переработок после проверки - пока не в первую очередь, не реализуем, потому что не могу придумать как именно реализовать).

- Данные относительно дня, недели, месяца.

Это прямой индикатор продуктивности. Сотрудники видят себя относительно других, не вы лично «гоните» — система показывает факт.

1.3. Разрез по партнёрам (клиентам)

- Всего заявок сегодня / обработано / осталось.

- По станам на «сьогодні»:
  - не роздруковані,
  - роздруковані, але не зібрані,
  - зібрані (Done),
  - прострочені.

- Скільки часу до дедлайну (мм:с), відображати колір:
  - зелений ≥ 60 хв,
  - жовтий < 60 хв,
  - червоний — прострочено.

- Waiting‑черга: кількість у `waiting`, «дальня дата» (найстаріше `scheduled_date`), «сьогодні створено у waiting».

- Хто сьогодні працював по цьому партнеру: список користувачів з лічильниками (замовлення/рядки/qty).

- Візуальний прапорець «потрібна увага» (червоний рядок), якщо до дедлайну мало часу або є прострочки.

- Время с распечатки до Done.

- Индикатор: ✅ успевает, ⚠️ риск, ❌ пропущен.

- Для «особых» партнёров — отдельная панель (например, те, у кого «большая заявка» или собственные критерии).

- Періоди: «сьогодні», «тиждень», «місяць» (перемикач замість жорсткого фільтра Today).

Это важно для оперативного управления сменой: кому срочно помочь, где перераспределить силы.

1.4. Приёмка (in-потоки)

- Количество принятых позиций.

- Время от создания до завершения приёмки.

- Время с распечатки до Done.

- Количество SKU, ожидающих размещения на полке.

- Данные относительно дня, недели, месяца.

Приёмка часто является «бутылочным горлышком», если задерживает обновление остатков — блокируется отгрузка.

2. Визуализация
2.1. Панель в модуле "Склад" (конкретика)

- Меню: Inventory → Operations → FLF Performance → Wallboard (полноэкранный режим, автообновление).
- Один экран Wallboard: сверху агрегированные показатели «Сегодня», ниже строки по партнёрам с прогресс-барами (✅/⚠️/❌), столбец риска, индикатор «большая заявка».
- ТОП‑списки сотрудников і партнерів:
  1) за кількістю: окремі рейтинги по рядках, по qty, по замовленнях;
  2) за швидкістю: наименьшее среднее время на одну заявку и на одну единицу;
  3) за педантичністю: найвищий % попадання в дедлайн.
  Періоди: день / тиждень / місяць.
- Блок «В зоне риска» (активные не-Done с дедлайном сегодня, сортированы по времени до дедлайна; для партнёров с расширенным окном также показывать попадание/непопадание в 18:30/19:00).
- Автообновление: базовый интервал конфигурируемый (по умолчанию 120 с), обновление рассинхронизированное по партнёрам/блокам (статтеринг), чтобы избегать всплесков нагрузки.
- Показывать у каждого блока подпись «Обновлено: HH:MM:SS».

- Multi‑company (службовий/технічний акаунт): виводити консолідовано по всім компаніям, до яких належить користувач; фільтр за компаніями залишити доступним.

- Блок «Отправки по слотам»: для каждого ближайшего слота машины показывать:
  - сколько заявок создано с момента предыдущего слота,
  - сколько успели отгрузить в этот слот,
  - сколько созданных не поехало (missed) — KPI для сравнения команд и сотрудников (меньше — лучше).

2.2. Визуальные сигналы

Цветовые индикаторы:

Зелёный — уложились в дедлайн.

Жёлтый — до дедлайна менее 60 мин.

Красный — нарушен дедлайн.

Прогресс-бар по каждому партнёру.

Таблица лидеров сотрудников (топ-5 по строкам/шт.) - относительно дня, недели, месяца. Например, лидер недели может получить небольшую дополнительную премию. Лидер месяца - премию побольше.

2.3. Персональная мини-панель (всегда под рукой, не мешает работе)

- Постоянный HUD‑стріп в верхней панели между systray‑виджетами и главным меню: всегда видимый блок с персональными метриками (мои строки, qty, среднее на строку, среднее print→done, мини‑индикатор дедлайна).
- Правая узкая панель в «Складе» — фиксированная (без сворачивания), компактный режим; подробности и история открываются по клику на элемент HUD или иконку systray.
- Показывать время последнего обновления; обновлять со сдвигом +10–20 с относительно Wallboard (статтеринг).

- Дополнительно: показатель «сколько созданных не поехало в слот» по моему закреплённому партнеру (primary partner). Если за партнёра отвечают несколько сотрудников, ответственность и метрика распределяются между всеми закреплёнными за партнёра.
 - Бейдж «Off‑day»: если сотрудник выполнял отгрузки в выходной (например, воскресенье), это учитывается как бонус и отображается отдельным бейджем с количеством таких отгрузок.

2.4. Панель руководителя

- Меню: Inventory → Reporting → FLF Manager Dashboard.
- Срезы за день/неделю/месяц, фильтры по партнёрам и сотрудникам; стандартные отчёты Odoo (pivot/графики) + быстрые KPI.
- Показатели: % в дедлайн, «всё созданное поехало сегодня», доли soft‑окна/последней машины/передачи смены, тренды ТОПов, индикатор «консолидация усилий» (позитивный сигнал координації команди).

2.5. Централизованные настройки (одно место)

- Меню: Inventory → FLF Performance → Configuration
- Разделы:
  - Settings — глобальные настройки через конфиг‑мастер (`res.config.settings` → `ir.config_parameter`):
    - Пороги «консолидации усилий» (X=40%, Y=30%, окно 14 дней), включение бейджа Off‑day.
    - Гибкая суббота: пороги бэклога (после 17:00 в пятницу) и утренней волны субботы; дефолтные часы начала смены по дням недели.
    - Интервал автообновления и базовый offset для статтеринга.
  - Partner SLA Registry — редактируемый список `res.partner` с FLF‑полями (дедлайн, soft window, last truck, afterhours, «большая заявка», draft‑триггер, ответственные пользователи).
    - Цель: управлять SLA по всем партнёрам из одного места, без перехода в форму партнёра.
  - User Assignments — список `res.users` с колонкой `x_flf_primary_partner_id` для удобного назначения «primary partner» (в т.ч. массово, позже через wizard).

- Доступ: для Stock Manager (Inventory) или отдельной группы «FLF Performance Manager», наследующейся от Stock Manager.
 - Доступ: конфигурация доступна ТОЛЬКО для «FLF Performance Manager» (наследуется от Stock Manager); одного Stock Manager недостаточно. Администратор Odoo имеет доступ всегда. Все параметры конфигурации — company‑dependent (настраиваются отдельно по компаниям).

3. Автоматические уведомления и реакции

Owl-виджет или toast-уведомление: за 30 мин до дедлайна партнёра появляется сообщение:
«До дедлайна по X Company осталось 30 мин. Обработано 78%, не завершено 5 заявок».

Telegram-бот или Odoo Discuss: уведомления для ответственного менеджера, если заявка перешла в «риск».

- Бонус‑уведомление по компании: «Сегодня всё созданное отправлено» (все созданные за сегодня заявки фактически уехали, с учётом графика машин).
 - Бонус‑уведомление для сотрудников, кто делал «Off‑day» отгрузки (вихідні/неділя) — для визнання додаткових зусиль.

4. Источники данных в Odoo (требует доосмысления и доработки)

stock.picking — основа (outgoing/incoming).

stock.move.line — строки (товары и qty).

stock.picking.printed — используем существующий флаг первого печатания (или точное время, если доступно) для расчёта метрик «print→done».

res.partner — хранение индивидуальных SLA и «особых» критериев. Настройка всех параметров по всем партнёрам — в одном месте (единый реестр/SLA-грід). Поля:

- x_deadline_hour — базовый дедлайн (час) партнёра. База расчёта — create_date.
- x_big_order_lines — порог строк «большая заявка» (дефолт 50).
- x_big_order_qty — порог qty «большая заявка» (дефолт 150).
- x_soft_window_minutes — расширенное «мягкое окно» после дедлайна (например, 90 минут до 18:30).
- x_last_shipment_time — время последней машины (например, 19:00).
- x_afterhours_credit — учитывать «передачу смены» (оформлено после работы, но до последней машины) как отдельный улучшенный статус по сравнению с полной просрочкой (по умолчанию True).
- x_refresh_offset_sec — опциональный сдвиг автообновления для партнёра (дробить нагрузку по времени).

Дополнительно:
- x_responsible_user_ids (Many2many res.users) — сотрудники, совместно отвечающие за этого партнёра.
- На пользователе: x_primary_partner_id (Many2one res.partner) — основной партнёр сотрудника (для персональных KPI та відповідальності).

Новые модели для графика машин:
- flf.truck.schedule — правило по дням недели и исключения (weekday, дата‑override, активность, примечание).
- flf.truck.slot — конкретные слоты отправки (в т.ч. будни 2–3 слота, по субботам иной график, по выходным — последний слот, напр. 17:00).

Настройки компании: глобальный график выезда машин (несколько слотов в день, различный по дням недели) — хранить как JSON или отдельной моделью. Используется для расчёта бонуса «всё созданное поехало сегодня».

Особенности выходных:
- В некоторые будни машины едут 2–3 раза.
- По субботам последний выезд в 17:00; дедлайны партнёров сдвигаются на 16:00.
- Воскресенье — обычно выходной; возможны исключения для срочных приёмок.
- Начало смены: в будни 09:00; по субботам при малой нагрузке 12:00, при высокой — 09:00 (может визначатися автоматично через «консолидацию усилий»).

Тогда логику легко стандартизировать и пересчитывать. Дедлайн рассчитываем от create_date; если не успели к x_deadline_hour, показываем отдельные индикаторы «уложился в мягкое окно (до 18:30)» и «оформлено до последней машины (19:00)»/«передача смены НП».

5. Мотивация и дисциплина (требует доработки)

Расчёт KPI за день / неделю / месяц.
Вычисление рейтинга сотрудников.

Формирование бонусов по критериям (норма, дедлайны, отсутствие ошибок).

Автоматическая публикация таблицы результатов ежедневно в Telegram.
 
 Бонусні показники: «Off‑day» отгрузки (вихідні/неділя) враховуються при підрахунку досягнень співробітника.

6. Рекомендации по ускорению

Динамика нагрузки. Если видите, что к 15:00 выполнено только 40% дневного плана — дайте системе сигнализировать.

Гибкая перераспределёнка. Панель должна позволять оперативно видеть, где загрузка минимальна — чтобы перекидывать людей.
