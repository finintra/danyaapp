[SLA партнерів] для Configuration → Partner SLA Registry:
Перелік партнерів з відхиленнями від дефолтів: Зроби 2 партнера з відхиленнями від дефолтів. У одного зроби раніше дедлайн. У іншого без дедлайну, але з ним будуть діяти додаткові правила: для великих замовлень:кількість оброблених рядків у день, кількість оброблених товарів у день. Для простих замовлень: має також все їхати день в день з дедлайном 17:30. Назови такого партнера "Electrinics LTD". 

Deadline hour (локальний час), Soft window (хв), Last truck hour (за партнера), After-hours credit (так/ні). - на твій розсуд, але поясни детально що це, бо не зовсім розумію.
«Велике замовлення»: ліміт ліній/кількості. - обговорювали раніше.
Draft triggers work (так/ні). - обговорювали раніше. 
Відповідальні користувачі (many2many). признач сам.

Пояснення полів SLA (локальний час компанії):
- Deadline hour — година, до якої замовлення повинні бути відправлені «день у день», якщо їх створено до цього часу (база — `create_date`).
- Soft window (хв) — «мʼяке вікно» після дедлайну, у межах якого відправка не вважається повною прострочкою (наприклад, +90 хв до 18:30).
- Last truck hour — час останньої машини (наприклад, 19:00). Якщо встигли до неї — краще, ніж «повна прострочка».
- After‑hours credit — якщо увімкнено, оформлення «передачі зміни» після роботи зараховується як проміжний кращий статус, а не «червона» прострочка.
- Draft triggers work — якщо «так», початок відліку для KPI беремо від створення чернетки; якщо «ні», рахуємо від переходу в Ready (Order Confirm).
- «Велике замовлення» — вважаємо великим, якщо перевищено поріг рядків або qty (використовується для пріоритизації/сигналів).

Приклади партнерів (створити у реєстрі SLA):
- Partner A (ранній дедлайн):
  - `x_flf_deadline_hour = 16`, `x_flf_soft_window_minutes = 60`, `x_flf_last_shipment_time = 19:00`, `x_flf_afterhours_credit = True`.
  - «Велике замовлення»: `x_flf_big_order_lines = 60`, `x_flf_big_order_qty = 180`.
  - Draft triggers work = False (рахуємо від Ready).
  - Відповідальні: Mitchell Admin + ще 1 тестовий користувач поточної компанії.
- Electrinics LTD (без фінального «жорсткого» дедлайну, але з додатковими правилами):
  - Для простих замовлень — «день у день» з дедлайном 17:30 (тобто `x_flf_deadline_hour = 17`, soft +30 хв).
  - Для великих — діють ліміти за день: поріг рядків = 80, поріг qty = 250 (рівні досягнення відслідковуються в KPI; можливі послаблення дедлайну для «великих»).
  - Draft triggers work = True (початок відліку від створення чернетки, щоб не затягували підготовку).
  - Last truck hour = 19:00, After‑hours credit = True.
  - Відповідальні: 2 тестові користувачі поточної компанії (призначити в many2many).

Ще важливе доповнення:
1. Якщо у компанії використовуються Draft як початок роботи з документом, тоді ми рахуємо що відлік часу починається з моменту створення Draft. Але якщо у компанії використовуються Order Confirm як початок роботи з документом, тоді ми рахуємо що відлік часу починається з моменту створення Order Confirm (тобто зі статусу Ready). І для таких компаній ми не звертаємо увагу на Draft взагалі, поки він не Ready. Бо це означає що працівники компанії просто ще у процесі створення замовлення (постачання чи відправка).
