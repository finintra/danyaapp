# Дедлайни, слоти та EOD (узгоджена модель)

Цей документ описує визначення «жорсткого» дедлайну (SLA), слотових порогів, «soft deadline», та розрахунок «EOD attention» — чи встигнемо до кінця робочого дня з поточними ресурсами.

## Визначення
- SLA‑дедлайн партнера (жорсткий):
  - Поле партнера: `x_flf_deadline_hour` (float, напр. 17.0 означає 17:00 локального часу партнера/користувача).
  - Системне поле пікингу: `x_flf_deadline_at` (UTC datetime). Обчислюється від `create_date` пікингу та години `x_flf_deadline_hour` із урахуванням TZ користувача.
  - Використання: індикатори «час до дедлайну», «In Deadline», «At Risk».

- Слотовий cutoff (час відправки машини):
  - Модельні сутності: `flf.truck.schedule` (правило дня/дати), `flf.truck.slot` (слоти `departure_time`).
  - Ідея: окремий «поріг хвилі», незалежний від SLA. Може використовуватись для KPI «скільки створених поїхало у слот / не поїхало (missed)».
  - Наступні кроки: зберігати розклад на рівні компанії, розраховувати слот‑метрики у KPI.

- Soft deadline (мʼяке вікно):
  - Поле партнера: `x_flf_soft_window_minutes` (int, хв). Це розширення до SLA (наприклад, +90 хв).
  - Інтерпретація: «не встигли у хвилю, але оформили у “передачу зміни” (НП: статус “на відділенні”)».
  - Використання: відображати окремим прапорцем у KPI/Wallboard (наприклад, `done_in_soft_window`).

## «At Risk» (поточний стан)
- Вмикається для активних не‑Done, коли до локального `x_flf_deadline_at` ≤ 60 хв (порог конфігурований).
- Для партнерів із Draft/Ready політикою «сьогоднішність» визначається як:
  - Draft‑тригер: за `create_date`.
  - Ready‑тригер: за `x_flf_ready_ts` (ставиться автоматично при переході у Ready).

## «EOD attention» — прогноз на кінець дня
Мета: показати по КОМПАНІЇ, чи встигнемо завершити сьогодні з наявними ресурсами.

- Змінні (company‑dependent):
  - `staff_on_shift` — кількість співробітників у зміні (шт).
  - `workday_end_hour` — локальний час завершення зміни (наприклад, 19:00).
  - `avg_order_base_min` — середній час на замовлення з одним товаром (хв), змінна А.
  - `avg_line_extra_min` — додатковий час на КОЖНУ позицію понад першу (хв), змінна B.

- Оцінка навантаження (workload):
  - Беремо сьогоднішні OUT у станах != done (Draft/Ready), з урахуванням політики старт‑події (див. вище).
  - Для кожного пікингу з `L` позиціями оцінка часу: `A + B * max(0, L - 1)`.
  - Сумарне навантаження, хв: `workload_min = Σ(A + B * max(0, L - 1))`.

- Оцінка спроможності (capacity):
  - `minutes_until_eod = max(0, (workday_end - now_local).minutes)`.
  - `capacity_min = staff_on_shift * minutes_until_eod`.

- Рішення:
  - Якщо `workload_min <= capacity_min` → «встигаємо сьогодні».
  - Якщо `workload_min > capacity_min` → «потребує уваги до EOD». Показники:
    - `eod_shortfall_min = workload_min - capacity_min` (дефіцит хв до кінця дня).
    - Орієнтовно «замовлень під ризиком» = `ceil(eod_shortfall_min / avg_order_effective_min)`; 
      де `avg_order_effective_min = A + B * avg(max(0, L - 1))` по активних замовленнях.

- Відображення:
  - На Wallboard Summary по компанії: індикатор «EOD attention», `eod_shortfall_min`, оцінка «замовлень під ризиком».
  - Без зміни `x_flf_deadline_at` (SLA залишається сталим), без привʼязки до конкретних замовлень.

## Нотатки по реалізації
- Усі параметри компанії зберігати через `res.config.settings` → `ir.config_parameter` (company‑dependent).
- Обчислення робити інкрементально у KPI‑cron (кожні 2 хв, конфігуровано), кешувати проміжні агрегати.
- Multi‑company: увага до часових поясів під час переведення `workday_end_hour` → UTC.
- Залежність від політики Draft/Ready: використовуємо відповідний «старт‑івент» (create_date vs ready_ts) для зрізу «сьогодні».
